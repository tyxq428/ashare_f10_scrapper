# 现金流量表主表与补充资料字段口径

东方财富 `RPT_F10_FINANCE_GCASHFLOW` 返回的单条报告期记录同时包含两组数据：

1. **现金流量表主表**：页面上方“经营活动、投资活动、筹资活动产生的现金流量”部分。
2. **补充资料**：页面下方“将净利润调节为经营活动现金流量”和“现金及现金等价物净变动情况”部分。

二者可能使用非常接近的中文项目名称，但原始 Key 不同：

| 页面语义 | 原始 Key | 披露特点 |
|---|---|---|
| 经营活动产生的现金流量净额（主表） | `NETCASH_OPERATE` | 一季报、中报、三季报、年报通常都有值 |
| 经营活动产生的现金流量净额（补充资料） | `NETCASH_OPERATENOTE` | 很多公司只在中报和年报披露，Q1/Q3为空是正常口径 |
| 现金及现金等价物净增加额（主表） | `CCE_ADD` | 主表字段 |
| 现金及现金等价物净增加额（补充资料） | `CCE_ADDNOTE` | 补充资料字段，可能稀疏 |

因此，不能用 `NETCASH_OPERATENOTE` 的空值判断页面主表中的经营现金流缺失。页面主表应读取 `NETCASH_OPERATE`。

## Excel 输出规则

从字段映射补丁 v2 起，Excel 将现金流量数据拆分为：

- `现金流量表`
- `现金流补充资料`
- `现金流量表同比`
- `单季度现金流`
- `单季度现金流补充资料`
- `单季度现金流同比`
- `单季度现金流环比`

所有原始 Key 均保留，不会删除补充资料字段；只是把不同披露口径分开显示，避免将补充资料的稀疏值误认为主表数据缺失。

## 数据完整性约束

每个请求组的 `record_count` 必须严格等于 `records` 数组长度。若缓存或中间文件出现例如：

```json
{
  "record_count": 64,
  "records": []
}
```

该请求组会被判定为不一致，不能复用，也不能生成最终 JSON、Excel、Parquet 或 DuckDB。
