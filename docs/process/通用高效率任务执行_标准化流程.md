# 通用高效率任务执行标准化流程

> 适用于数据整理、研究分析、文件处理、自动化开发、批量网页检索、报告生成、模型调用、系统排障等复杂任务。
>
> 核心目标不是“完成一次”，而是：**可规划、可执行、可观测、可恢复、可验证、可复用，并在执行过程中持续提高单位时间内的有效完成率。**

---

## 1. 总体原则

### 1.1 先定义“完成”，再开始执行

任务开始前必须明确：

- 目标是什么；
- 输入是什么；
- 输出是什么；
- 哪些来源允许使用；
- 哪些操作禁止；
- 什么状态算完成；
- 什么状态只能算部分完成；
- 什么情况必须人工介入。

不得使用“先做着看”的方式启动复杂任务。

### 1.2 以有效完成率为核心指标

推荐主指标：

```text
单次完整率提升 = 本轮新增通过验收的项目数 / 全部目标项目数
```

辅助指标：

```text
有效吞吐量 = 本轮新增通过验收的项目数 / 执行时间
一次通过率 = 首次执行即通过验证的项目数 / 本轮处理项目数
返工率 = 需要重新处理的项目数 / 已处理项目数
缓存复用率 = 直接复用已有结果的步骤数 / 总步骤数
自动化率 = 无需人工判断完成的项目数 / 总项目数
阻塞率 = 因环境、权限或数据缺失无法推进的项目数 / 总项目数
```

不能只看“处理了多少”，必须看“新增多少可靠、可交付、通过验证的结果”。

### 1.3 优先批量处理，不按对象逐个手工推进

一般优先顺序：

1. 可直接批量复制或映射的内容；
2. 可通过公式或规则批量计算的内容；
3. 可从统一结构中批量提取的内容；
4. 可通过同一来源横向处理的内容；
5. 需要专项检索的异常项；
6. 需要人工判断的少数尾部项。

### 1.4 成功一轮后自动继续

除非出现以下情况，不应因为取得少量进展而暂停：

- 已完成全部任务；
- 环境无法继续执行；
- 缺少必要权限或私有数据；
- 出现不可自动消解的来源冲突；
- 需要用户作出实质性口径选择；
- 即将触发资源限制，继续执行会导致成果丢失。

## 2. 四层执行架构

### 2.1 Prompt：传递本次任务变量

Prompt只承载本次执行特有的信息：

- 输入文件或数据；
- 目标范围；
- 来源限制；
- 输出格式；
- 优先级；
- 时间范围；
- 验收标准。

不要在每次Prompt中重复大量固定流程。

### 2.2 Skill：固化执行政策和工作流

Skill适合固化：

- 何时触发；
- 固定的前置检查；
- 标准执行顺序；
- 数据来源优先级；
- 状态标签；
- 失败恢复规则；
- 质量检查清单；
- 阶段性输出条件。

Skill是控制层，不应堆积大量一次性业务数据。

### 2.3 MCP或连接器：提供稳定的工具接口

MCP适合提供：

- 文件搜索、读取和写入；
- 报告下载与缓存；
- 数据库查询；
- 网页检索；
- PDF解析；
- Excel读取、计算和回写；
- 任务状态读取和保存；
- 日志、指标和错误报告。

MCP负责“能力”，Skill负责“如何组合能力”。

### 2.4 Scripts：执行确定性和高风险步骤

脚本适合：

- 环境预检；
- 批量下载；
- 数据清洗；
- 格式转换；
- 公式计算；
- 唯一键生成；
- 重复项检测；
- 结果合并；
- 文件验证；
- 断点恢复。

能够确定性实现的步骤，不应每次依赖模型临场推理。

## 3. 标准执行流程

## 阶段0：任务分级与模式选择

执行前先判断任务属于哪一类：

### A. 简单任务

特征：

- 单一输入；
- 步骤少；
- 无外部依赖；
- 失败成本低。

策略：直接执行并验证。

### B. 中等任务

特征：

- 多步骤；
- 需要文件或联网；
- 有一定计算或格式要求。

策略：先形成短计划，执行薄切片验证，再批量推进。

### C. 复杂任务

特征：

- 多文件、多来源、多工具；
- 长时间运行；
- 需要批量处理；
- 有来源约束；
- 失败会造成大量返工。

策略：必须启用完整SOP、缓存、快照、日志、验证和恢复机制。

### D. 高风险任务

特征：

- 涉及资金、法律、医疗、安全、生产环境或不可逆修改。

策略：增加审批点、只读预演、双重验证和回滚方案。

## 阶段1：需求标准化

将用户自然语言要求转换为“任务合同”。

### 3.1 必填字段

```yaml
objective: 最终目标
scope: 处理范围
inputs: 输入文件、数据、账号或系统
outputs: 输出文件、结构、格式和位置
allowed_sources: 允许使用的来源
forbidden_sources: 禁止使用的来源
constraints: 时间、成本、权限、格式、准确率限制
acceptance_criteria: 完成标准
priority_metric: 优先优化的指标
human_intervention_rules: 人工介入条件
```

### 3.2 消除歧义

重点确认：

- “完整”是结构完整、事实完整，还是数值完整；
- “核验”是找到来源，还是来源直接支持结论；
- “没有”是明确披露为零，还是未检索到；
- “最新”对应哪个截止日期；
- “全部”是否包含无法公开取得的数据；
- 是否允许派生计算；
- 是否允许使用第三方镜像或二手来源。

### 3.3 建立验收矩阵

| 项目 | 验收条件 | 不合格示例 |
|---|---|---|
| 完整性 | 所有目标项均有明确状态 | 空白、待定 |
| 准确性 | 数值和结论与来源一致 | 推测值冒充披露值 |
| 来源 | 可追溯到具体文件和位置 | 只有搜索结果标题 |
| 计算 | 公式、单位和口径正确 | 年度和单季度混用 |
| 可恢复 | 有最新快照和重试队列 | 中断后需从头开始 |
| 可交付 | 输出文件可打开且结构正确 | 文件损坏、链接不存在 |

## 阶段2：基线审计

在修改任何内容前建立基线。

### 4.1 输入审计

检查：

- 文件是否存在；
- 文件格式是否正确；
- 文件是否损坏；
- 编码和语言；
- 表头和字段；
- 数据行数；
- 唯一对象数；
- 重复项；
- 已完成项；
- 占位项；
- 公式和引用；
- 隐藏表、冻结窗格、合并单元格等结构。

### 4.2 建立唯一键

唯一键必须能稳定定位一个任务单元，例如：

```text
对象ID | 字段 | 时间口径 | 期间 | 单位 | 版本
```

唯一键用于：

- 去重；
- 增量更新；
- 断点恢复；
- 冲突检测；
- 合并结果。

### 4.3 建立初始快照

快照至少包含：

```json
{
  "task_id": "任务唯一标识",
  "input_hash": "输入文件哈希",
  "total_items": 0,
  "completed_items": 0,
  "pending_items": 0,
  "blocked_items": 0,
  "failed_items": 0,
  "status_by_type": {},
  "last_successful_step": "",
  "next_action": "",
  "retry_queue": [],
  "artifacts": [],
  "updated_at": ""
}
```

## 阶段3：环境预检

### 5.1 文件与权限

检查：

- 工作目录是否可读写；
- 输入文件是否可访问；
- 输出目录是否有空间；
- 文件名和路径是否包含不兼容字符；
- 是否存在覆盖原文件的风险；
- 是否需要先复制只读备份。

### 5.2 运行环境

检查：

- Python、Shell或运行时版本；
- 必要依赖；
- 工具版本；
- CPU、内存、磁盘；
- 单次执行超时；
- 最大并发限制；
- 可用上下文长度；
- 是否支持长任务持续运行。

### 5.3 网络环境

检查：

- DNS；
- HTTPS访问；
- 目标站点可用性；
- 代理；
- 证书；
- 速率限制；
- 登录或验证码；
- 下载文件是否被反爬拦截。

### 5.4 工具能力矩阵

| 能力 | 首选工具 | 备用工具 | 最后手段 |
|---|---|---|---|
| 文件搜索 | 文件连接器 | 本地索引 | 人工定位 |
| 网页检索 | Web工具 | 浏览器/MCP | 人工提供链接 |
| PDF文本 | 预处理文件服务 | pdftotext | OCR |
| Excel写回 | 专用表格工具 | Python库 | 手工修改 |
| 长任务 | 脚本＋快照 | 分批执行 | 人工拆分 |

### 5.5 环境路由

预检后必须明确选择：

- `full_local`：本地完成全部步骤；
- `split_mode`：联网工具获取，本地脚本处理；
- `connector_mode`：主要依赖MCP或连接器；
- `manual_gate`：必须人工提供权限或数据后继续。

## 阶段4：制定执行计划

### 6.1 建立任务依赖图

将任务拆分为：

- 独立步骤；
- 前置步骤；
- 可并行步骤；
- 关键路径；
- 高风险步骤；
- 人工决策点。

### 6.2 选择最高效率的处理维度

通常按以下顺序评估：

1. 按字段横向处理；
2. 按来源横向处理；
3. 按文档章节横向处理；
4. 按规则横向处理；
5. 按对象逐个处理。

只有当对象之间差异极大时，才优先按对象处理。

### 6.3 估算收益和成本

为每个工作包估算：

```text
预计完整率提升
预计耗时
预计失败率
是否可自动化
是否可复用缓存
来源质量
```

优先级可用如下评分：

```text
优先级 = 预计完整率提升 × 来源可靠性 × 自动化成功率 / 预计耗时
```

### 6.4 设定批次门槛

建议：

- 小任务：一轮完成；
- 大任务：每批至少提升10%完整率；
- 大型表格：每批至少新增100个通过验收的任务单元；
- 尾部阶段：允许按剩余全部项目一次性完成。

---

## 阶段5：薄切片试运行

正式大批量执行前，选择一个具有代表性的最小样本。

薄切片应覆盖：

- 一个正常案例；
- 一个缺失数据案例；
- 一个异常格式案例；
- 一个需要计算的案例；
- 一个需要来源判断的案例。

验证：

- 输入读取是否正确；
- 字段映射是否正确；
- 单位和口径是否正确；
- 输出格式是否保持；
- 缓存和日志是否生成；
- 失败是否进入重试队列。

薄切片通过后，才扩大批次。

---

## 阶段6：高吞吐执行

### 8.1 一次加载，多次复用

对于同一个文件、网页或报告：

- 只下载一次；
- 只解析一次；
- 只建立一次索引；
- 多个字段复用同一缓存。

### 8.2 缓存层次

建议缓存：

```text
L1：当前运行内存缓存
L2：本地解析结果缓存
L3：原始文件缓存
L4：任务级结果缓存
L5：跨任务可复用知识或索引
```

缓存键应包含：

```text
来源URL或文件哈希 + 版本 + 解析器版本 + 目标范围
```

### 8.3 幂等执行

重复运行时：

- 不重复处理已成功项目；
- 不重复下载未变化文件；
- 不覆盖人工确认结果；
- 仅处理新增、变化或失败项；
- 输出结果与执行次数无关。

### 8.4 并发原则

适合并发：

- 独立网页下载；
- 独立文件解析；
- 独立对象检索；
- 独立计算。

不适合直接并发：

- 同一Excel文件写入；
- 共享状态无锁更新；
- 具有速率限制的接口；
- 依赖顺序的计算。

采用“并发读取、集中写入”的模式。

### 8.5 结构化中间结果

研究、解析和写回之间不要直接传递自然语言，应优先使用结构化记录：

```json
{
  "key": "对象|字段|期间",
  "value": null,
  "unit": "",
  "status": "FACT_DIRECT",
  "source": "",
  "source_date": "",
  "location": "",
  "calculation": "",
  "confidence": "high",
  "notes": ""
}
```

## 阶段7：执行监控与瓶颈优化

### 9.1 每轮必须记录的指标

- 本轮处理数；
- 本轮新增通过数；
- 单次完整率提升；
- 执行时间；
- 平均单项耗时；
- 失败数；
- 重试成功数；
- 来源命中率；
- 缓存命中率；
- 返工率。

### 9.2 瓶颈分类

#### A. 环境瓶颈

表现：命令失败、网络不通、依赖缺失、文件无法写入。

优化：

- 切换工具；
- 分离联网和本地处理；
- 提前安装依赖；
- 调整输出目录；
- 减小批次；
- 增加超时和重试。

#### B. 数据瓶颈

表现：公开来源没有数据、口径不一致、数据被付费墙阻挡。

优化：

- 明确记录NA而不是猜测；
- 寻找替代指标；
- 扩大来源类型；
- 请求人工提供私有数据；
- 降低结论粒度。

#### C. 算法或规则瓶颈

表现：同类任务仍需逐条手工判断。

优化：

- 提炼字段映射；
- 编写解析规则；
- 使用模板和正则；
- 建立异常词典；
- 将计算固化为脚本。

#### D. 工具调用瓶颈

表现：频繁打开同一文件、重复搜索、调用次数过多。

优化：

- 批量查询；
- 多关键词一次检索；
- 一次读取更大连续范围；
- 建立本地索引；
- 合并工具调用。

#### E. 上下文瓶颈

表现：长任务中遗忘规则、重复研究、输出不一致。

优化：

- 使用状态JSON；
- 将固定规则写入Skill；
- 将数据放入外部文件；
- 每轮只加载相关引用；
- 保留简短运行摘要。

#### F. 尾部瓶颈

表现：前90%很快，最后10%极慢。

优化：

- 将尾部按阻塞原因分类；
- 单独建立异常处理器；
- 明确哪些必须人工；
- 避免用低质量推断凑100%。

### 9.3 自适应优化循环

每轮结束后执行：

```text
测量 → 识别瓶颈 → 修改策略 → 小样本验证 → 扩大执行
```

如果连续两轮的有效吞吐量下降超过30%，必须停止当前方法，重新规划，而不是继续堆叠低效操作。

## 阶段8：断点、日志与恢复

### 10.1 检查点触发条件

- 完成一个字段组；
- 完成一个对象组；
- 达到固定时间间隔；
- 达到固定新增完成数；
- 执行高风险操作前；
- 可能触发超时前；
- 切换工具或执行环境前。

### 10.2 必须保存的内容

- 当前结果文件；
- 状态快照；
- 原始缓存；
- 解析缓存；
- 错误日志；
- 重试队列；
- 未解决项；
- 下一步操作；
- 版本和运行参数。

### 10.3 错误分类

```text
RETRYABLE：临时网络、限流、超时
DATA_ERROR：输入格式、缺失字段、单位异常
LOGIC_ERROR：规则或计算错误
SOURCE_CONFLICT：不同来源冲突
PERMISSION_BLOCKED：登录、授权、验证码
HUMAN_DECISION：需要用户选择口径
```

### 10.4 重试规则

- 临时网络错误：指数退避重试；
- 404或永久缺失：立即切换备用来源；
- 格式异常：进入异常解析队列；
- 逻辑错误：停止同类批处理，修复后重跑受影响部分；
- 权限错误：不反复重试，直接标记人工介入。

---

## 阶段9：质量验证

### 11.1 四层验证

#### 第一层：结构验证

- 行数、列数正确；
- 表头未变化；
- 唯一键无重复；
- 格式和样式保留；
- 输出文件可打开。

#### 第二层：数据验证

- 单位一致；
- 时间口径一致；
- 数值类型正确；
- 空值状态合理；
- 极端值和负值经过检查。

#### 第三层：逻辑验证

- 合计关系；
- 同比、环比、占比；
- 期末与期初关系；
- 派生公式；
- 不重复计算；
- 不跨口径推断。

#### 第四层：来源验证

- 来源真实可访问；
- 来源支持对应结论；
- 日期和期间匹配；
- 原文位置明确；
- 直接披露和计算结果分开标记；
- 未检索到不标记为明确为零。

### 11.2 抽样与全量验证

- 结构和公式错误：全量验证；
- 唯一键和重复：全量验证；
- 高风险数值：全量复核；
- 普通文本提取：分层抽样；
- 自动规则首次上线：扩大抽样；
- 发现系统性错误：回溯全部同类结果。

### 11.3 交叉验证

对于重要结果：

- 尽量使用两个独立来源；
- 或使用原始来源＋计算复核；
- 或使用前后期间关系验证。

## 阶段10：人工介入规则

只有满足以下条件才暂停并请求人工介入：

1. 缺少登录、验证码或付费权限；
2. 所需数据属于公司内部或私有系统；
3. 权威来源之间存在无法自动判定的冲突；
4. 用户必须选择业务口径；
5. 操作不可逆且风险较高；
6. 环境不支持必要工具；
7. 继续执行只能依靠猜测或低质量推断。

人工介入请求必须包含：

```text
受阻项目：具体哪些任务单元
已经完成：完成到什么程度
已尝试：使用了哪些来源和方法
根因：为什么自动化无法继续
所需人工动作：用户最少需要做什么
恢复入口：人工动作完成后从哪个快照继续
影响范围：不介入会缺失哪些结果
```

不得只说“无法完成”或“请提供更多信息”。

## 阶段12：复盘与持续改进

每次复杂任务结束后记录：

- 哪些步骤最耗时；
- 哪些步骤重复执行；
- 哪些错误可以提前发现；
- 哪些规则适合写入Skill；
- 哪些动作适合写成脚本；
- 哪些能力适合做成MCP工具；
- 哪些资料应长期缓存；
- 哪些字段需要新增数据源；
- 哪些人工判断可以模板化。

将改进分为：

### Prompt层

补充本次任务变量、约束和验收标准。

### Skill层

补充固定流程、决策规则、质量标准和停止条件。

### Script层

将重复、确定、易错的动作自动化。

### MCP层

将外部系统和稳定能力封装为可复用工具。

### 数据层

积累来源索引、字段字典、单位映射、异常案例和缓存。

## 6. 标准继续执行Prompt

```markdown
从最新检查点继续执行。

要求：
1. 不重新处理已成功项目；
2. 优先处理能够使完整率提升最大的剩余工作包；
3. 先分析上一轮失败和耗时分布，优化后再执行；
4. 成功完成一轮后自动继续下一轮；
5. 只有确实需要权限、私有数据或业务口径选择时才暂停；
6. 输出前保存最新结果、快照、日志和重试队列；
7. 最终报告新增完成数、累计完整率、剩余项及真实瓶颈。
```

## 7. 成熟度分级

### L0：临时执行

无计划、无缓存、无验证、无法恢复。

### L1：有计划

有步骤和输出目标，但仍依赖人工推进。

### L2：可恢复

有快照、日志、唯一键和失败重试。

### L3：可批量

具备字段优先、批量规则、缓存和集中写入。

### L4：自适应

根据实时指标识别瓶颈并调整策略。

### L5：平台化

Prompt、Skill、MCP、脚本和数据资产协同，可跨任务复用。

长期目标应达到L4，重复频繁的任务应逐步建设到L5。

---

## 8. 最重要的十条执行纪律

1. 不定义完成标准，不开始。
2. 不做环境预检，不批量运行。
3. 不建立唯一键，不做增量更新。
4. 不保存快照，不执行长任务。
5. 不验证薄切片，不扩大批次。
6. 不重复下载、解析或搜索相同内容。
7. 不把未找到写成零或不存在。
8. 不因小进展频繁暂停。
9. 不在效率持续下降时沿用原方案。
10. 不在无法可靠完成时用推测凑完整率。
