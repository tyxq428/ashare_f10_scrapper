# 688521 数据输出与仓库代码基线审计

> 审计类型：只读基线审计  
> 代码基线：`main@6997c1a6894b2b9414aa196bbd631e9dbb2ce8b9`  
> 数据样本：`688521_official_full(1)(1).xlsx`、`688521_cross_validation(1)(1).xlsx`  
> 本文只记录问题、边界与改进优先级，不包含代码修改。

## 1. 执行摘要

当前仓库已经从“F10抓取器”发展为具备以下能力的平台：

- 113 个固定东方财富请求组；
- JSON、Excel、Parquet、DuckDB 多格式输出；
- 年度、季度、单季度、TTM 和安全公式计算；
- 上交所及深交所/CNINFO 免费官方报告发现与下载；
- PDF 财务表解析；
- 字段验证注册表；
- 双源比较、会计逻辑检查和 TTM 双公式检查；
- Raw Pack 原文缓存、哈希、实体匹配和权限状态路由；
- 检查点、缓存和 E2E 工作流。

这已经解决了“是否有官方源、能否找到报告、能否把部分核心财务项目与 F10 对上”的第一阶段问题。

但从输出审计看，当前系统仍处在：

> **字段分类覆盖较高，但官方事实解析覆盖低；被成功比较的数值准确性较好，但解析器会生成少量高风险伪事实；Raw Pack、官方验证和研究映射尚未形成统一证据图。**

最重要的结论不是“0个冲突”，而是：

1. 被两边成功取到且可比较的 548 项没有发现冲突；
2. 仍有 43,443 条 `MISSING_OFFICIAL` 和大量未加载历史期间；
3. 官方解析中已经发现一类系统性假阳性，必须先修复后再扩大覆盖；
4. 当前的 `comparable_match_rate=1.245%` 混合了覆盖和准确性，容易被误读。

## 2. 仓库现有架构审计

### 2.1 已有优势

| 能力 | 当前实现 | 应保留的设计 |
|---|---|---|
| F10基础抓取 | 固定、版本化的113个请求组 | 固定清单、请求去重、失败精确回退 |
| 缓存与恢复 | 原始响应缓存、检查点、增量重试 | 幂等与只处理失败/变化项 |
| 官方发现 | SSE/CNINFO报告发现，排除摘要，优先更正版 | 版本优先级和正式文件筛选 |
| PDF证据 | 页码、原始行、文档URL和哈希 | 事实级来源穿透 |
| 字段分类 | 3,330个字段上下文100%进入某种验证模式 | “不适用”和“需未来来源”不强行匹配 |
| 会计逻辑 | 资产负债、现金流和TTM检查 | 确定性验证与生成链路分离 |
| Raw Pack | 来源清单、实体匹配、权限阻塞、NO_MATCH不作负面推断 | 来源状态纪律与原文留存 |
| 输出 | Excel/Parquet/DuckDB/JSON | 机器可读与人工审阅并行 |

### 2.2 当前关键模块关系

```text
东方财富接口
  -> FetchPipeline
  -> combined.json / facts.parquet / f10.duckdb / F10 Excel

官方定期报告
  -> SSEOfficialSource / CNInfoOfficialSource
  -> PdfStatementParser
  -> official facts

F10 facts + official facts
  -> FieldValidationRegistry
  -> CrossSourceComparator
  -> logic checks / TTM checks
  -> cross-validation exports

Raw Pack
  -> 独立 source manifest / collectors / parsers
  -> 原始网页和来源索引
```

主要结构性缺口是最后一层：**Raw Pack 与官方事实解析、规范事实映射、双源比较是三条相邻但未统一的数据链。**

### 3.3 发现的P0正确性缺陷：附注号误识别为金额

样本中存在：

```text
字段：DEFER_TAX_ASSET / 递延所得税资产
生成值：29
原始行：递延所得税资产 七、29

字段：DEFER_TAX_LIAB / 递延所得税负债
生成值：29
原始行：递延所得税负债 七、29
```

这里的“七、29”是报表附注编号，不是金额。由于 PDF 表格换行或金额位于相邻行，当前解析逻辑把唯一剩余数字当成了金额。

该问题的风险：

- 生成了不存在的官方事实；
- 事实被标记为 `FACT_DIRECT` 且置信度为 `high`；
- 因 F10 未找到对应值，被归类为 `MISSING_EASTMONEY`，没有触发数值冲突；
- “0个真正冲突”因此无法发现该错误；
- 若该事实进入研究映射包，会污染资产负债、税率、估值或盈利质量分析。

该问题说明官方解析器必须增加：

1. 附注编号识别；
2. 跨行/跨单元格表格行重建；
3. 金额列位置约束；
4. 解析可疑状态；
5. 事实级 sanity check；
6. 固定回归样本。

### 3.4 官方事实模型仍需增强

| 当前问题 | 影响 | 建议 |
|---|---|---|
| `field_category` 全部为 `PAGE_DISPLAY_FIELD` | 无法表达财务、元数据、派生、事件等语义 | 使用研究本体分类 |
| 所有直接事实置信度均为high | 解析器无法表达模糊/可疑事实 | 根据表格结构、金额列、别名和逻辑检查评分 |
| 本地路径包含运行目录名称 | 数据包搬移后不稳定 | 使用包内相对路径和document_id |
| 缺少解析器/目标注册表/本体版本 | 不能稳定重放和缓存失效 | 写入版本与哈希 |
| 缺少bbox/table_id/line_span | 难以精准定位跨行错误 | 增加版面证据定位 |
| 比率名称部分仍使用英文key | 人工审阅不友好 | 本体提供规范中文名 |
| 年报元数据同时出现FY和Q4派生行 | 容易混淆报告类型与独立季度 | 将报告元数据与独立季度派生事实分层 |
| URL部分为HTTP | 规范性和长期可用性较弱 | 保存规范HTTPS URL和原始URL |

### 4.3 指标语义问题：1.245%不是准确率

当前 `comparable_match_rate` 的分母包含：

- 已成功找到两边值并比较的事实；
- 官方解析未找到的事实；
- 官方仅有/F10仅有事实；
- 部分解析/路由覆盖缺口。

因此1.245%主要反映“官方报告和解析覆盖尚低”，不是被比较数据的准确率。

更合理的指标体系：

| 指标 | 定义 |
|---|---|
| 分类覆盖率 | 有明确验证模式的字段上下文 / 全字段上下文 |
| 报告加载覆盖率 | 已加载目标报告期 / 应加载目标报告期 |
| 文档解析成功率 | 成功解析文档 / 已下载文档 |
| 目标字段提取覆盖率 | 提取到的官方目标字段 / 预期目标字段 |
| 双源可比覆盖率 | 两边均有可比较事实 / 目标F10事实 |
| 双源匹配准确率 | 匹配事实 /（匹配事实+真实冲突） |
| 官方仅有比例 | 官方有、F10无 / 官方事实 |
| F10仅有比例 | F10有、官方目标源未找到 / 目标F10事实 |
| 证据完整率 | 带文档、页码、原始行、哈希的事实 / 官方事实 |

按当前样本，成功完成真正双源比较的事实没有发现冲突，因此“比较准确率”接近100%；但“可比覆盖率”很低。这两个结论必须同时展示。

### 4.4 字段分类存在粗粒度误路由

审计发现部分市场派生时间序列和平台口径字段被归入 `OFFICIAL_DOCUMENT_EVENT`，例如：

- 股息率/分红影响曲线；
- 收盘价、PE、PB等行情派生值；
- 某些股东或持仓序列；
- 平台生成的排名和聚合指标。

这些字段的正确处理应分别是：

- 行情或平台口径：`EASTMONEY_SOURCE_SPECIFIC`；
- 可由官方分红与市场价格计算：`CALCULATED_MARKET_FACT`；
- 官方公告事件：用精确公告路由和事件唯一键验证；
- 机构/持仓平台统计：保留来源特定属性，不强行与定期报告比对。

当前按“family包含某模式”分类，足以达到100%分类率，但不足以保证分类正确率。

### 4.5 状态体系不能区分缺口根因

当前 `MISSING_OFFICIAL` 可能代表完全不同的问题：

1. 官方报告没有披露；
2. 目标报告没有加载；
3. 解析器不支持该章节；
4. 解析器支持但本次解析失败；
5. 事实应从临时公告而非定期报告获取；
6. 事实仅属于第三方平台口径；
7. 文档存在更正版本但未加载；
8. 事实为明确零但表格以横线表示；
9. 官方事实被解析为可疑值，不应参与比较。

应拆为明确状态：

- `DOCUMENT_NOT_LOADED`；
- `SOURCE_ROUTE_NOT_IMPLEMENTED`；
- `PARSER_COVERAGE_GAP`；
- `PARSE_SUSPECT`；
- `NOT_DISCLOSED_IN_SOURCE`；
- `NOT_APPLICABLE_TO_SOURCE`；
- `FACT_ZERO_EXPLICIT`；
- `FUTURE_FREE_SOURCE_REQUIRED`；
- `PERMISSION_BLOCKED`；
- `SOURCE_CONFLICT`；
- `UNRESOLVED`。

### 4.6 只加载两个报告期不够支撑研究映射包

当前样本加载：

- 2025年年度报告；
- 2026年第一季度报告。

它适合作为薄切片验证，但研究映射包至少需要可选的：

- 最近5年年报；
- 最近8—12个季度；
- 更正/修订版本链；
- IPO招股书及问询回复（上市时间较短时）；
- 关键临时公告；
- 股本与资本事项文件。

建议提供两种明确预设：

- `thin-slice`：最近年报+最新季报，快速验证；
- `research-full`：五年财务+最近季度+关键事件文档。

## 5. 代码层重点问题

### 5.1 PDF解析器

高风险点：

- `_choose_amounts()`在只剩附注号时仍可能选择该数字；
- 金额选择主要依据数值大小、逗号、小数和括号，不理解表格列位置；
- 跨页/跨行续表重建不足；
- 解析结果缺少可疑置信度；
- 解析成功与事实正确没有分离。

### 5.2 单位处理

旧验证入口通过 `math.nextafter(1.0, 2.0)` 区分“明确单位：元”和“未找到单位默认元”。该做法数值上影响极小，但属于隐式哨兵值：

- 可读性差；
- 容易在未来浮点或序列化处理中产生隐蔽问题；
- 单位来源不应编码进scale浮点值。

应改为显式对象：

```text
unit=元
scale=1.0
unit_origin=PAGE_EXPLICIT | ROW_EXPLICIT | INHERITED | DEFAULT
```

### 5.3 比较容差

当前数值比较先使用统一的 `abs(difference) <= 1.0`：

- 对亿元金额可能过严或无意义；
- 对每股指标、比率、百分比、股数和人数可能过宽；
- 对不同小数位和原始单位不够精确。

应由指标本体或官方事实提供：

- `absolute_tolerance`；
- `relative_tolerance`；
- `display_decimals`；
- `unit_scale`；
- `comparison_method`。

### 5.4 Point-in-time缺口

官方发现使用当前日期作为查询终点，CLI没有`as_of_date`：

- 历史研究回放可能使用后来更正或后来发布的文件；
- 缓存和输出没有明确研究截止日；
- 无法严格避免未来函数。

### 5.5 缓存键不完整

解析缓存目前主要由：

```text
document_sha256 + parser_cache_version
```

决定。若动态目标字段、字段别名或研究本体变化，旧缓存可能被错误复用。建议加入：

```text
document_sha256 + parser_version + target_registry_hash + ontology_version + parser_options_hash
```

### 5.6 Raw Pack与官方验证未统一

Raw Pack已经有来源分层、实体匹配、权限阻塞和原始网页保存，但：

- 定期报告解析器有另一套文档模型；
- 交叉验证只使用官方财务报告；
- F10公告ID没有统一作为官方原文发现种子；
- 原文证据、规范事实和比较结果缺少统一document_id/fact_id。

## 6. 测试缺口

现有测试已覆盖：

- 更正版报告优先；
- 基础数字解析；
- 报表章节识别；
- 会计逻辑；
- TTM公式。

优先新增：

1. “递延所得税资产 七、29”不得产生29金额；
2. 跨行金额重建；
3. 明确横线/不适用/零的状态；
4. 合并与母公司范围冲突；
5. 累计与独立季度冲突；
6. 截止日后更正版本不得回填；
7. 单位来源与单位继承；
8. 指标级容差；
9. 市场派生字段不得路由为官方报告事件；
10. 覆盖率与准确率指标定义；
11. 证据路径搬移后仍可访问；
12. 缓存随目标注册表变更失效。

## 7. 优先级矩阵

| 优先级 | 问题 | 严重性 | 原因 |
|---|---|---:|---|
| P0 | 附注号误识别为金额 | 极高 | 产生伪官方事实且被标高置信度 |
| P0 | 无as-of-date控制 | 极高 | 历史回测和研究快照可能未来函数 |
| P0 | 覆盖率与准确率混为一体 | 高 | 可能误读系统质量 |
| P0 | 指标/单位容差不精确 | 高 | 产生假冲突或假匹配 |
| P0 | 字段模式误路由 | 高 | 大量不适用比较和错误缺口 |
| P1 | Raw Pack与证据链不统一 | 高 | 无法形成真正研究级证据包 |
| P1 | 缺少规范事实本体和多对一血缘 | 高 | 同一事实跨接口重复，研究层难消费 |
| P1 | 仅两个报告期 | 中高 | 适合薄切片，不适合完整首次覆盖 |
| P1 | 状态根因不够细 | 中高 | 缺口不可行动、自动恢复困难 |
| P2 | 单位浮点哨兵 | 中 | 技术债和可维护性风险 |
| P2 | 证据路径不便携 | 中 | 项目包移动后引用失效 |
| P2 | 测试和性能指标不足 | 中 | 难以安全扩大覆盖 |

## 8. 与现有并行PR的边界

### PR #18：SSE上市前历史期间覆盖

- 本项目不重复实现其报告发现修复；
- 后续实现阶段先检查其合并状态；
- 若已合并，直接基于新的报告发现结果构建研究包；
- 若未合并，本项目保持适配器接口兼容并避免修改同一文件，必要时先同步main再继续。

### PR #16：可视化执行中心

- 本项目先定义稳定输出契约和状态指标；
- UI只消费最终契约，不在本项目早期反向驱动底层模型；
- 可视化对接放在数据契约和E2E稳定之后。

## 9. 基线结论

当前系统的正确定位是：

> **已具备可靠的F10基础采集、官方报告发现和双源验证框架；下一阶段必须优先提升官方事实解析正确性和研究级数据契约，而不是继续横向增加更多接口。**

代码修改的第一批不应是“大规模扩字段”，而应是：

1. 修复伪事实；
2. 建立可疑解析闸门；
3. 拆分覆盖与准确性指标；
4. 增加as-of-date；
5. 校正字段路由；
6. 再建设规范研究映射和统一证据图。
