# SSE上市前报告期覆盖与全历史对账修复计划

## 任务合同

- **目标**：解释并修复`688521`全报告期双源验证中“缺少大量官方报告”和84条来源冲突的问题。
- **输入**：东方财富完整事实表、上交所免费正式披露、上交所免费公司/上市信息、已缓存PDF及解析缓存。
- **输出**：更准确的报告期生命周期状态、可追溯的官方覆盖明细、无伪冲突的全历史对账包、测试与使用文档。
- **允许来源**：上交所、科创板官方页面和正式披露文件；不使用收费数据库。
- **禁止事项**：不得把未找到的上市前报告写成0；不得把上市前非适用期间记为普通“报告缺失”；不得为消除冲突而隐藏真实差异。
- **完成标准**：
  1. 每个请求报告期都归入`已发现定期报告 / 上市前期间 / 上市后真实缺失 / 已发现但解析失败`之一；
  2. 688521上市日期有可追溯来源；
  3. 上市前期间不再进入普通`missing_report_dates`；
  4. 2020Q3和2021Q1等已发现但零提取文档有独立状态并尽量修复；
  5. 全历史84条冲突逐条分类，伪冲突修复后仅保留可证明的真实冲突；
  6. JSON、Excel、Parquet、DuckDB、Web指标与状态定义一致；
  7. 全量测试、688521最近两期回归、688521全历史E2E、002352 CNINFO回归、Raw Pack回归均通过。

## 基线判断

688521于2020-08-18在科创板上市。当前上交所定期报告接口仅查询`DQBG`（定期报告），因此2016年至2019年以及2020Q1/H1等上市前期间本来就不会以该上市公司代码存在于上交所定期报告库。东方财富中的这些历史财务记录可能来自招股说明书、申报文件或后续年报的比较数据，而不是对应期间的上市公司定期报告。

当前输出混合了三种不同问题：

1. **上市前不适用**：没有对应上市公司定期报告，不应写成普通缺失；
2. **已发现但解析为0**：例如部分早期报告版式，需要解析器修复；
3. **真正的上市后报告发现缺口**：查询范围内按规则应存在但未发现，需要重试或替代官方入口。

84条`MISMATCH`必须从Artifact逐条审计，不能仅凭数量判断为真实来源冲突。

## 执行阶段

### 阶段1：诊断快照

1. 使用当前`main`代码运行688521全历史双源任务；
2. 输出请求日期、发现文档、零提取文档、84条冲突的字段/期间/来源行；
3. 保存诊断JSON、Parquet和官方文档清单；
4. 对冲突按单位、符号、累计/单季、重述版本、表内同名项目、解析错行分类。

### 阶段2：证券生命周期解析

1. 新增统一`SecurityLifecycle`模型；
2. 优先从上交所公司概况公开接口读取上市日期；
3. 备用从东方财富基本资料字段读取，并明确记录`listing_date_source`；
4. 为每个报告期计算：
   - `POST_LISTING_PERIODIC_EXPECTED`
   - `PRE_LISTING_PERIOD`
   - `LISTING_TRANSITION_PERIOD`
5. 将生命周期信息写入summary、DuckDB和Excel。

### 阶段3：报告发现状态重构

将现有`missing_report_dates`拆分为：

- `periodic_report_dates_available`
- `post_listing_missing_report_dates`
- `pre_listing_report_dates`
- `discovered_but_zero_extraction_dates`
- `pre_listing_alternative_source_dates`
- `unresolved_report_dates`

新增对账状态：

- `PRE_LISTING_PERIOD_NOT_APPLICABLE`
- `PRE_LISTING_OFFICIAL_SOURCE_NOT_LOADED`
- `OFFICIAL_DOCUMENT_EXTRACTION_FAILED`

这些状态不计入真正来源冲突。

### 阶段4：上市前免费官方替代来源

1. 接入上交所官方公司概况/科创板公司快照，验证可公开提供的2018—2020年度核心财务数据；
2. 调研并接入上交所发行上市审核/招股说明书公开文件发现；
3. 对招股说明书中明确披露的历史年度或中期财务事实生成`OFFICIAL_PRE_LISTING_DISCLOSURE`事实；
4. 仅在文档明确给出对应期间时参与比较，未披露季度不推算；
5. 保存文档URL、SHA-256、页码和原始行。

### 阶段5：零提取文档修复

1. 针对2020Q3、2021Q1等已下载但提取为0的PDF检查文本层、表格标题、单位和扫描属性；
2. 扩展标题、续表、跨行和表格坐标规则；
3. 扫描件仅在确实必要时进入人工/OCR状态，不静默判为缺失。

### 阶段6：84条冲突修复

1. 使用完整唯一键：证券代码、报告期、期间类型、报表类型、合并范围、字段Key、报告版本、单位；
2. 修复可证明的解析错行、单位、正负号、累计/单季、原版/更正版问题；
3. 同一官方事实验证多个东方财富端点时保留端点级记录，但统一官方事实身份；
4. 真实来源差异继续保留`MISMATCH`并要求证据，不以白名单掩盖。

### 阶段7：展示和导出

1. Web增加“上市日期”“上市前期间”“上市后真实缺失”“发现但解析失败”指标；
2. Excel增加：
   - `报告期生命周期`
   - `上市前替代来源`
   - `零提取文档`
   - `冲突原因分类`
3. `PASS_WITH_COVERAGE_GAPS`必须附带缺口分类，不再只显示一个总数。

### 阶段8：测试与验收

- 单元测试：生命周期边界、上市前状态、报告发现分类、冲突分类；
- 薄切片：688521最近两期仍保持0真实冲突；
- 全历史：所有请求期间都有明确状态，上市前期间不再列为普通报告缺失；
- 回归：002352巨潮、300308现金流/行情、Raw Pack、任务中心、XLSX多列布局；
- Artifact重新打开验证JSON、Excel、Parquet、DuckDB和证据ZIP。

## 并行与恢复策略

- 报告发现、公司概况查询、PDF缓存审计可并行；
- PDF只下载一次，按`URL + SHA256 + parser_version`复用；
- 先提交本计划，再提交诊断快照，再分批提交生命周期、来源和解析修复；
- 每次失败只重跑受影响步骤；
- 临时Workflow和诊断文件在最终PR前删除。

## 人工介入条件

仅在以下情形暂停：

- 官方文件需要验证码/登录；
- 上市前招股说明书是扫描件且文本/表格无法可靠解析；
- 官方原版与更正版之间存在无法自动确定的生效关系；
- 84条冲突中出现两个权威官方文件数值不一致且无法依据版本规则解决。
